Ocrypto - Function Overview

Function Overview

Algorithms

This version of ocrypto supports the following algorithms:

Algorithms Specification PQC-Resistant
Message digests (hashes)
SHA2 (224, 256, 384, 512) FIPS-180-4 yes 1)
Message authentication codes (MAC)
HMAC (SHA256, SHA512) RFC2104 yes 1)
AES-CMAC RFC4493, NIST SP 800-38B yes 2)
Unauthenticated ciphers
AES (CTR, ECB, CBC) FIPS-197, NIST SP 800-38* yes 2)
Authenticated encryption with associated data (AEAD)
AES (GCM) FIPS-197, NIST SP 800-38* yes 2)
Key wrap
AES KW/KWP NIST SP 800-38F yes 2)
Asymmetric signature
ECDSA / Deterministic ECDSA (P-256) FIPS-186-5 no
Key agreement
ECDH (P-256) NIST SP 800-56A no

1) Use of SHA384/SHA512 recommended.

2) Use of AES-256 recommended.

Key Sizes

  • AES: 128, 192, and 256 bit keys are supported for all uses of AES.
  • NIST P-256 curve (aka secp256r1) supports 256 bit keys.

Function Types

ocrypto supports these two types of functions:

Single-part

Function is executed in a single call, using only the stack.

Incremental

Function is executed in multiple calls, to allow for streams of data to be processed in a piecemeal fashion. Additional memory is provided by caller in a context object. The functions are split into sets of functions whose names end in init, final and update.

There are no stack-optimized functions in this distribution of ocrypto.

All functions are available as single-part function types.

RAM Requirements

ocrypto does not use static memory (no global variables).

ocrypto does not use dynamic memory allocation - no heap is required.

The following table indicates how much memory is used on the stack:

Function and type Required size (bytes)
Single-part < 1,000
Incremental < 300

For incremental functions, the caller must provide pre-allocated context object, of similar size as stack space for single-part functions. The application may allocate these context objects statically or in a heap.
